# Design: Pre-Prod Regression (Promptfoo + Call-Driver Harness)

Implements [[Finding 2]], [[Finding 9]], [[Finding 10]], and [[Finding 11]].
Owns: Promptfoo scenario packs, the eval-tenant/Square-Sandbox test
environment, the call-driver harness that actually places calls, and CI
release-gate wiring. Calls into [02](02-voice-evaluation-layer.md) for all
scoring — this doc only orchestrates.

## Hard dependency — confirm before building anything here

This entire component is blocked on a backend-team ticket from
[[Finding 9]]: registering a dedicated `eval-` prefixed test tenant whose
`bot_config` points to Square **Sandbox** credentials instead of a real
merchant account. This session's codebase check confirmed **zero** existing
sandbox/test-tenant concept anywhere in `tenant_service.py` or
`squareup_service.py` — no contradiction of `idea.md`'s prior finding, but
also no evidence this ticket has been filed or completed yet. **Verify

and Square Sandbox read-back have nothing to run against.

## 1. Scenario packs

Promptfoo config organized by tenant, per `design.md`'s scenario schema:
`scenarios/{tenant_id}/*.yaml` feed into a per-tenant
`promptfooconfig.yaml`. Promptfoo's native `python` assertion type calls
directly into `voiceeval` ([[Finding 10]]):

```yaml
# promptfooconfig.yaml (sketch)
providers: [...]   # the call-driver harness, see §3 below
tests:
  - vars:
      scenario_file: scenarios/eval-iscream-gelato/add_two_scoops_001.yaml
    assert:
      - type: python
        value: file://assertions/order_accuracy.py
      - type: python
        value: file://assertions/menu_price_rubric.py
```

Each `assertions/*.py` file is a thin shim: load the scenario YAML, call the
matching `voiceeval` function, return Promptfoo's expected pass/fail or
score shape. No scoring logic lives in the shim itself.

## 2. The call-driver harness

[[Finding 11]] identified that Promptfoo cannot place or drive a real
Ultravox call on its own — something has to actually make the call. This
session's research **partially resolves** the mechanism, in a way that's
more concrete than [[Finding 11]]'s original framing:

**Key insight: unlike [01](01-observability.md)'s passive listener, the
harness *is* the call creator.** It calls the existing `POST /calls/web`
flow against the `eval-` tenant itself, so it receives the `joinUrl`
synchronously in the response — no push/pull discovery problem, because
there's no third-party-joins-someone-else's-call issue here at all. This
removes one entire class of risk that [01](01-observability.md)'s listener
still has to spike.

**Proposed flow:**

1. Harness calls `POST /calls/web` for the `eval-` tenant (same endpoint
   the live product already uses), gets back a `joinUrl` (`wss://...`).
2. Harness connects to that `joinUrl` as a WebSocket client, acting as the
   "customer" side of the session.
3. Harness drives the persona: for `mode: scripted` (the thin-slice
   default, per [[Finding 2]]), it synthesizes each script line via TTS and
   sends the resulting audio over the connection at the appropriate point
   in the turn-taking cycle (using `state` messages to know when the agent
   has finished speaking and is `listening`). **This step is not yet
   buildable as described** — the audio frame format/codec Ultravox
   expects on this send path is the Open Spike below; treat "synthesizes
   and sends" as the intended shape of the step, not a confirmed mechanism,
   until that spike is closed.
4. Harness captures the agent's audio output and Data Connection transcript
   deltas for the duration of the call — the same kind of event stream
   [01](01-observability.md)'s listener consumes, but generated by a call
   we created ourselves.
5. On call end, harness hands the captured transcript/audio + the scenario's
   `expected_order` to [02](02-voice-evaluation-layer.md)'s
   `score_order_accuracy()` and `judge_rubric()`, and (for the test tenant
   specifically) triggers the Square Sandbox read-back below.

### Open spike — not yet resolved

The exact audio frame format/codec Ultravox expects when a server-side
WebSocket client sends synthesized speech into a session has not been
confirmed — `docs.ultravox.ai/apps/websockets` exists and almost certainly
documents this, but its contents were not read in this session's research.
**This is an implementation-detail spike** (codec, sample rate, framing),
not a structural unknown like [01](01-observability.md)'s join-permission
question — resolve it by reading that doc directly before writing the
harness's audio-sending code.

### `llm_simulated` persona mode

Deferred past the thin slice ([[Finding 2]]). When picked up: an LLM plays
the customer role, reacting to the agent's actual responses rather than
following a fixed script — gives broader edge-case coverage at the cost of
non-determinism (a regression test that doesn't reproduce identically is a
weaker signal). Start with `scripted` only; revisit once the scripted path
is proven and there's an appetite for the added non-determinism.

## 3. Square Sandbox read-back

Per `idea.md`'s Finding 7 addendum: since Square's order-creation response
is never persisted by the backend (`order_details` is always empty —
`ultravox_webhooks.py:906`), the harness itself must call Square Sandbox's
GET-order API directly after a test call completes, using the test
tenant's Sandbox credentials only (never production Square read access).
This becomes the `actual_order` input to `score_order_accuracy()` for
test-tenant calls specifically (per [02](02-voice-evaluation-layer.md)'s
ground-truth table) — it independently verifies the integration layer, not
just the agent's self-reported understanding.

## 4. CI / release gate

Verified this session: `echobot-studio-backend`'s CI is GitHub Actions.

- `.github/workflows/deploy.yml` — triggered on push to `main` or manual
  dispatch; runs `sam build` → `sam deploy` directly, **no test/lint gate
  today**.
- `.github/workflows/deploy-reusable.yml` — a separate reusable workflow
  that already runs `just lint build` (a `Justfile` recipe doing `sam
  validate --lint` + build) before deploying — this is the existing
  pre-deploy hook point, but `deploy.yml` itself doesn't call it.

**Proposed insertion point:** add a `just eval-gate` recipe (running
`promptfoo eval --config scenarios/.../promptfooconfig.yaml` with a
threshold-based exit code) into `deploy-reusable.yml` alongside the
existing `just lint build` step, so it runs before `sam deploy` wherever
that reusable workflow is invoked. `deploy.yml`'s direct path would still
bypass it unless that workflow is also updated to route through the
reusable one — **this is backend-repo CI configuration, so wiring it in is
backend-team-coordinated work**, same ticket-filing pattern as
[[Finding 4]]/[[Finding 9]], not something this project can land
unilaterally.
